## 六、增值税电子专用发票特殊性说明

### 6.1 该类型的独特特性

1. **电子发票介质**
   - 以电子数据形式开具、传输和存储
   - 无需纸质打印，法律效力等同纸质发票
   - 通过OFD（开放版式文档）格式或PDF格式提供
   - 可通过二维码或电子签章验证真伪

2. **无校验码**
   - 电子专用发票**没有校验码**字段（JYM为空）
   - 与纸质专票相同
   - 防伪验证通过电子签章和二维码实现

3. **可抵扣进项税**
   - 与纸质专票法律效力完全相同
   - 一般纳税人取得电子专票可按票面税额抵扣
   - 无需打印，可直接使用电子文件进行税务申报

4. **必须有货物明细**
   - 电子专票必须包含货物或应税劳务明细
   - CHILDLIST不能为空
   - 每个明细行需要包含完整的税率、税额信息

5. **无税控设备编号**
   - 电子发票开具无需传统税控设备
   - 长软API中的JQBH（机器编号）字段通常为空
   - 开具依赖电子税务局或税务UKey

6. **与纸质专票的主要区别**
   - **发票介质**: 电子 vs 纸质
   - **税控设备**: 无需税控盘/金税盘 vs 需要税控设备
   - **存储方式**: 电子文件存储 vs 纸质保管
   - **传输方式**: 电子传输（邮件、系统对接） vs 纸质流转
   - **其他要素**: 发票代码、号码、购销方信息、金额、税率等完全相同

7. **金额计算关系**
   ```
   价税合计 = 金额合计 + 税额合计
   税额合计 = Σ(明细金额 × 明细税率)
   金额合计 = Σ(明细金额)
   ```

### 6.3 特殊处理逻辑

#### 6.3.1 电子发票文件管理

**说明**：虽然映射表关注查验接口，但业务系统需要妥善管理电子发票文件。

**建议**：
- 存储OFD或PDF格式的原始电子发票文件
- 记录文件的SHA256哈希值，确保文件完整性
- 建立电子档案管理系统，便于检索和备查
- 保留电子签章信息，用于离线验证

#### 6.3.2 金额计算验证

与纸质专票相同，需要验证：
- 价税合计 = 金额合计 + 税额合计（允许±0.02误差）
- 明细金额合计与汇总一致（允许±0.02误差）
- 明细税额合计与汇总一致（允许±0.02误差）
- 每个明细的税额 = 金额 × 税率（允许±0.01误差）

#### 6.3.3 税率识别

从明细中提取税率的方法：
- 遍历items数组，提取所有非零税率
- 优先选择最常见的税率作为主税率
- 如果税率一致，直接取该税率
- 如果有多种税率（混合销售），可选择加权平均或最高税率

常见税率：13%、9%、6%、0%、免税

#### 6.3.4 电子签章验证（扩展功能）

**说明**：长软查验接口不返回电子签章信息，如需验证签章，需从电子发票文件中提取。

**验证要素**：
- 电子签章颁发机构（CA机构）
- 签章有效期
- 签章完整性（是否被篡改）
- 签章与发票内容的绑定关系

- `invoice_status` (发票状态) - 从HEAD/CYJGDM映射，必填
- `seller_tax_no` (销售方识别号) - 必填
- `issue_date` (开票日期) - 必填

### 7.2 字段长度差异

| 字段 | 本公司长度 | 长软长度 | 实际风险评估 |
|------|-----------|---------|------------|
| invoice_number | 20 | 10 | ✅ 本公司长度更大，兼容数电票号 |
| seller_tax_no | 20 | 26 | ⚠️ 长软更大，但中国税号18位，20位足够 |
| buyer_tax_no | 20 | 64 | ⚠️ 长软更大，但中国税号18位，20位足够 |
| name | 310 | 320 | ⚠️ 长软略大，截断风险极小 |

### 7.3 数据完整性

**长软独有字段**（本公司API无对应）:
- `CYCS` (查验次数) - 长软统计查验次数，可记录但不返回
- `CYSJ` (查验时间) - 长软查验时间戳，可记录但不返回
- `JQBH` (机器编号) - 税控设备编号，电子专票通常为空
- `CPYBZ` (成品油标志) - 已废止，以TSPZBZ为准
- `QDBZ` (清单标志) - 是否有销货清单，可扩展字段支持
- `HZDK` (汇总代开标志) - 是否汇总代开，可扩展字段支持

**本公司独有字段**（需从其他数据源获取）:
- `void_date` (作废日期) - 从业务系统获取
- `proxy_seller_*` (代开信息) - 从业务规则解析
- `tax_rate` (主税率) - 从明细计算得出
- `item_count` (明细条数) - 从items数组长度计算
- `tax_inclusive_rate_flag` (含税税率标识) - 从业务规则判断
- `applicable_tax_rate_flag` (适用税率标识) - 从业务规则判断
- `non_taxable_amount` (不征税金额) - 从业务系统获取
- `seller_taxpayer_type_code` (销方纳税人类型) - 从业务系统获取
- `vehicle_abnormal_flag` (机动车异常标识) - 从业务系统获取（电子专票一般为0）
- `issue_type` (开票类型) - 仅乐企查验返回

### 7.4 增值税电子专用发票特殊集成要点

⚠️ **这是最重要的部分，必须根据业务特性详细说明！**

#### 7.4.1 查验请求参数构建

**与纸质专票的区别**：
- **发票类型映射**: 本公司08 → 长软20（纸质专票01 → 长软01）
- **其他查验要素**: 完全相同（发票代码、号码、开票日期、不含税金额）

**请求参数示例**：
```
本公司API请求:
{
  "invoice_type": "08",
  "invoice_code": "044001900111",
  "invoice_number": "12345678",
  "issue_date": "2025-12-30",
  "invoice_amount": 10000.00
}

映射到长软API:
<MSG>
  <VERSION>4.0.12</VERSION>
  <FPLX>20</FPLX>
  <FPDM>044001900111</FPDM>
  <FPHM>12345678</FPHM>
  <KPRQ>20251230</KPRQ>
  <FPJE>10000.00</FPJE>
  <JYM></JYM>
  ...
</MSG>
```

#### 7.4.2 电子发票文件存储建议

虽然查验接口只返回发票要素，但业务系统应考虑：
1. **原始文件保存**: 保存OFD或PDF格式的电子发票文件
2. **文件关联**: 将查验结果与电子发票文件建立关联（通过发票代码+号码）
3. **备份策略**: 定期备份电子发票文件，防止数据丢失
4. **检索优化**: 建立索引，支持按发票代码、号码、日期、购销方等条件快速检索

#### 7.4.3 与数电发票的区分

**易混淆点**：
- **电子专票(08)**: 传统电子发票，有12位发票代码，长软类型20
- **数电专票(81)**: 全电发票，无发票代码，20位发票号码，长软类型09(QDLX=20)

**区分规则**：
- 有发票代码（12位） → 传统电子专票(08)
- 无发票代码，发票号码20位 → 数电专票(81)

#### 7.4.4 明细数据处理

与纸质专票处理完全相同：
- 验证明细金额、税额合计与汇总一致
- 从明细中提取主税率
- 支持混合销售（多种税率）
- 验证每个明细的税额计算

#### 7.4.5 电子签章和二维码

**说明**：
- 长软查验接口**不返回**电子签章和二维码信息
- 如需验证签章和二维码，需要从原始电子发票文件中提取
- 建议在业务系统中保存签章验证结果
