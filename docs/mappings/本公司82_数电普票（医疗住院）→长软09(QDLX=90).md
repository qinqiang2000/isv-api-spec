## 📋 概述

本文档定义了本公司发票查验API与长软发票查验服务之间，针对"数电普票（医疗住院）"的完整字段映射关系。

- **本公司API**: [`/partners/invoice-verifications`](https://s.apifox.cn/2795ad32-31e7-4363-8294-578e8dc6202c/395314528e0) (RESTful JSON格式)
- **长软API**: 【长软】发票查验服务调用说明V4.1.2.pdf (XML格式)
- **本公司发票类型代码**: 82
- **长软发票类型代码**: 09 (数电发票)
- **长软数电发票类型**: QDLX=90 (医疗-住院)
- **长软文档章节**: 3.2.3.20
- **长软API版本**: V4.1.2（新增）

### ⚠️ 重要：触发条件

**本映射表适用于以下情况**：
1. 当本公司发票类型为82（数电普票）
2. **且**满足以下任一条件：
   - 特殊票种标志 `TSPZBZ = 15`（医疗服务-住院）
   - 明细中包含医疗住院特征（项目名称包含"住院"、"床位费"等）
   - 长软查验结果返回 `QDLX = 90`
3. **且**查验请求中 `REQTYPE = V2`

**映射路由逻辑**：
```
如果 invoice_type = 82:
    如果 TSPZBZ = 15 或 长软返回QDLX=90:
        如果 REQTYPE = V2:
            使用本映射表（82→09(QDLX=90)）
        否则:
            使用标准数电普票映射表（82→09(QDLX=10)）
    否则:
        使用标准数电普票映射表（82→09(QDLX=10)）
```

## 三、货物/明细字段映射

### 3.1 明细结构说明

⚠️ **关键架构决策**：数电医疗住院发票（09_90）的明细数据需要**拆分为两个独立数组**：

**本公司API结构**：
```json
{
  "items": [],                           // 标准货物明细（保持82数电普票的语义）
  "medical_inpatient_detail_list": []    // 医疗住院特定业务要素（新增字段）
}
```

**长软API路径**: `BODY/CHILDLIST/CHILD[]`（单一数组）

⚠️ **为什么必须拆分？**
1. **语义正确性**：`items[]`的字段（如`specification`=规格型号）不能被重用为项目名称，会导致字段语义混乱
2. **数据一致性**：保持82类型发票的`items[]`结构统一，无论是普通数电普票还是医疗住院数电普票
3. **业务扩展性**：医疗住院特定信息（项目名称、费用明细、商品编码）应该放在专门的业务字段中
4. **参考通行费发票**：类似于通行费发票（72）的拆分架构

**拆分逻辑**：
- 长软09_90的每个`CHILD`元素同时映射到两个数组
- `items[]`：保留标准货物字段（金额、税率、税额），其他字段（规格型号、单位等）留空
- `medical_inpatient_detail_list[]`：包含医疗住院特定信息（项目名称、备注、商品编码等）

## 四、数据格式转换说明

### 4.1 日期格式转换

| API | 格式 | 示例 |
|-----|------|------|
| 本公司API | `YYYY-MM-DD` 或 `YYYY-MM-DD HH:mm:ss` | 2025-12-31 或 2025-12-31 16:59:45 |
| 长软API | `YYYYMMDD` | 20251231 |

**转换规则**：
- 开票日期 (KPRQ): `YYYYMMDD` → `YYYY-MM-DD`
- 实现: `${yyyymmdd.substr(0,4)}-${yyyymmdd.substr(4,2)}-${yyyymmdd.substr(6,2)}`

### 4.2 数值类型转换

| 数据项 | 本公司API | 长软API | 转换说明 |
|--------|-----------|---------|----------|
| 金额 | 本公司从items计算 | `VARCHAR2(32)` 字符串 | 长软JE字段存在但本公司不使用 |
| 税额合计 | `number` (18,2) | `VARCHAR2(32)` 字符串 | 长软→本公司: `parseFloat()` |
| 价税合计 | `number` (18,2) | `VARCHAR2(32)` 字符串 | 长软→本公司: `parseFloat()` |
| 明细金额 | `number` (8,2) | `VARCHAR2(32)` 字符串 | 长软→本公司: `parseFloat()` |
| 明细税额 | `number` (8,6) | `VARCHAR2(32)` 字符串 | 长软→本公司: `parseFloat()` |
| 实际税额 | - | `VARCHAR2(32)` 字符串 | 长软独有字段 |

**转换示例**：
```javascript
// 长软API返回
{
  "JE": "1000.00",       // 金额（字符串）
  "SE": "0.00",          // 税额（字符串，医疗免税）
  "JSHJ": "1000.00"      // 价税合计（字符串）
}

// 转换为本公司API
{
  "total_tax_amount": 0.00,
  "total_amount": 1000.00,
  "amount_with_tax_in_words": "壹仟元整"
}
```

## 六、数电医疗住院发票特殊性说明

### 6.1 该类型的独特特性

1. **数电化的医疗住院发票**
   - 这是传统医疗发票的数电化版本
   - 无发票代码，仅有20位发票号码
   - 无校验码，防伪通过数字签名实现
   - 通过电子税务局开具

2. **V4.1.2新增类型**
   - **QDLX=90**：专门用于医疗住院发票
   - **REQTYPE参数**：必须设置为V2才能返回此格式
   - **向后兼容**：如果REQTYPE=V1或不填，医疗发票仍按照09_10格式返回

3. **与普通数电普票(82→09_10)的区别**
   - **长软类型**: 09(QDLX=10)用于普通数电普票，09(QDLX=90)专用于医疗住院
   - **明细字段**: 09_10为常规货物明细，09_90为医疗项目明细
   - **特殊票种标志**: 09_90有TSPZBZ=15字段，09_10可能没有
   - **状态字段**: 都使用ZFBZ

4. **明细数据拆分架构**
   - 长软09_90的明细数据需要**拆分为两个独立数组**
   - `items[]`：保持标准货物明细结构（规格型号、单位等留空）
   - `medical_inpatient_detail_list[]`：存储医疗住院特定业务要素（项目名称、备注、商品编码等）
   - ❌ **禁止字段重用**：不能将`specification`重用为项目名称，会导致语义混乱

5. **医疗服务免税**
   - 医疗住院服务通常享受免税政策
   - 税额合计(SE)通常为0
   - 价税合计(JSHJ) = 金额合计(JE)
   - TSZCBS通常标识为1（免税）或04（免税）

6. **四方信息简化**
   - 长软返回的地址电话、开户行账号字段通常为空
   - 数电票简化了四方信息
   - 购方主要填写名称和识别号

7. **医疗住院特有字段**
   - `XMMC`（项目名称）：记录医疗服务项目（如"床位费"、"治疗费"）
   - `BZ`（备注）：记录项目详细说明
   - `SPBM`（商品编码）：医疗服务税收分类编码
   - `TSZCBS`（特殊政策标识）：通常为免税
   - `SJSE`（实际税额）：实际缴纳税额

#### 6.3.5 金额计算验证

```javascript
// 验证价税合计 = 金额合计 + 税额合计
function validateMedicalInvoiceAmount(items, totalTax, totalAmount) {
  // 计算明细金额汇总
  const itemsAmount = items.reduce((sum, item) => sum + item.amount, 0);
  const itemsTax = items.reduce((sum, item) => sum + item.tax_amount, 0);

  // 验证明细汇总
  if (Math.abs(itemsAmount - (totalAmount - totalTax)) > 0.02) {
    console.warn("明细金额汇总与汇总金额不一致");
  }

  if (Math.abs(itemsTax - totalTax) > 0.02) {
    console.warn("明细税额汇总与汇总税额不一致");
  }

  // 医疗免税发票验证
  if (totalTax === 0) {
    if (Math.abs(itemsAmount - totalAmount) > 0.02) {
      throw new Error("免税发票价税合计应等于金额合计");
    }
  } else {
    // 验证价税合计
    if (Math.abs((itemsAmount + itemsTax) - totalAmount) > 0.02) {
      throw new Error("价税合计验证失败");
    }
  }

  return true;
}
```

```json
}
```

**拆分方案的优势**：
1. ✅ **语义正确性**：每个字段名都准确描述其实际含义
2. ✅ **数据一致性**：所有82类型发票的`items[]`结构保持统一
3. ✅ **可维护性**：未来添加新字段时不会产生歧义
4. ✅ **API清晰性**：调用方不会因为字段重用而困惑
5. ✅ **业务扩展性**：可以灵活扩展医疗住院特有字段，不影响标准货物明细

**实现要点**：
1. 在查验响应中**必须包含**`medical_inpatient_detail_list`字段（针对09_90类型）
2. `items[]`和`medical_inpatient_detail_list[]`通过`sequence_no`一一对应
3. 两个数组的金额、税额应保持一致（用于验证）
4. API文档中明确说明`medical_inpatient_detail_list`仅在医疗住院类型发票中出现

**数据验证**：
```javascript
// 验证两个数组的数据一致性
function validateMedicalInpatientDataConsistency(items, medicalInpatientDetailList) {
  if (items.length !== medicalInpatientDetailList.length) {
    throw new Error("items和medical_inpatient_detail_list长度不一致");
  }

  items.forEach((item, index) => {
    const detail = medicalInpatientDetailList[index];

    if (item.sequence_no !== detail.sequence_no) {
      throw new Error(`序号不匹配：items[${index}]=${item.sequence_no}, detail[${index}]=${detail.sequence_no}`);
    }

    if (Math.abs(item.amount - detail.amount) > 0.01) {
      throw new Error(`金额不一致：序号${item.sequence_no}`);
    }

    if (Math.abs(item.tax_amount - detail.tax_amount) > 0.01) {
      throw new Error(`税额不一致：序号${item.sequence_no}`);
    }
  });

  return true;
}
```

#### 7.4.3 免税政策判断

**挑战**：医疗住院服务通常享受免税政策，需要正确处理。

**解决方案**：
1. **默认规则**：
   - 医疗住院发票（09_90类型）通常免税
   - 税额合计(SE)为0
   - 价税合计(JSHJ) = 金额(JE)

2. **检查特殊政策标识**：
   - 检查明细中的 `TSZCBS`（特殊政策标识）
   - 如果所有明细都是免税(1/04) → 确认免税
   - 设置税率为0

3. **业务规则补充**：
   - 如果业务系统有明确的免税规则，优先使用业务规则
   - 建议在查验响应中添加免税标识字段

#### 7.4.4 医疗机构识别

**挑战**：需要识别销售方是否为医疗机构。

**解决方案**：
1. **通过特殊票种标志**：
   - 如果TSPZBZ=15，确认为医疗住院发票
   - 销售方必然是医疗机构

2. **通过税号识别**：
   - 医疗机构通常为事业单位或企业
   - 税号格式符合标准18位统一社会信用代码

3. **通过名称关键字**：
   - 名称包含"医院"、"卫生院"、"诊所"等关键字
   - 建议在业务系统中维护医疗机构白名单

#### 7.4.5 向后兼容处理

**挑战**：如何保证旧系统（使用V1或不设置REQTYPE）的兼容性。

**解决方案**：
1. **默认行为**：
   - 如果请求中不包含REQTYPE或REQTYPE=V1
   - 医疗发票仍按照09_10格式返回
   - 旧系统不受影响

2. **逐步迁移**：
   - 新系统统一使用REQTYPE=V2
   - 旧系统保持REQTYPE=V1或不设置
   - 根据业务需求逐步迁移

3. **数据兼容**：
   - 09_10格式的医疗发票可以转换为09_90格式
   - 主要差异在于明细字段的拆分
   - 建议在中间层做数据转换
